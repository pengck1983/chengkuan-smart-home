# 成宽智慧小家项目复盘与技术总结

## 项目目标

项目从单一 MQTT 调试页面逐步演化为可登录、可绑定网关、可管理多设备、支持微信小程序和小智 AI 双入口的完整物联网原型。核心目标不是重新制造一套硬件协议，而是把现有 Zigbee 网关、EMQX、业务服务和用户交互可靠地连接起来。

## 演进过程

第一阶段完成小程序直接通过 MQTT over WebSocket 连接 EMQX，验证插座控制和状态上报。开发中遇到 Node.js `stream` 依赖不适配微信运行时，最终改用小程序兼容的 MQTT.js 构建版本。

第二阶段引入 Node.js 业务服务器。微信登录、用户与网关绑定、设备归一化和历史记录迁移到服务端，小程序不再持有 MQTT 密码。cpolar 提供可供微信访问的 HTTPS 入口。

第三阶段重构小程序体验：设备卡片、详情页、插座定时控制、环境趋势、彩灯模式和“智家/我的”双标签。不同账号可以绑定不同网关，为多用户扩展奠定基础。

第四阶段接入小智 AI MCP。Python Bridge 将自然语言工具调用映射到 Node 内部 API，避免 Python 再次连接 MQTT。随后修复注册应答、数值类型、设备在线判定、计量字段和灯光模式等问题。

## 关键技术决策

1. 使用 EMQX 作为消息骨干，解耦网关与应用。
2. 使用 Node.js 作为唯一设备业务真相源，前端和 AI 共用同一状态模型。
3. 采用公网业务 API与本机内部 MCP API双端口设计。
4. 前端不保存基础设施凭据，登录后只持有短期业务 Token。
5. 对厂商协议增加归一化层，而不是让 UI 理解原始字段。

## 典型问题与经验

- “MQTT 已连接”不等于“网关已注册”。注册 ACK 缺失会导致指示灯异常和状态链路不完整。
- 零值必须与缺失值区分，环境温度显示 0 往往来自默认值或错误倍率。
- 类型边界要显式处理。JSON 中的字符串数值进入格式化函数后，直接使用数值格式符会抛异常。
- 控制请求不能只看 publish 成功，应结合响应 Topic 或后续状态确认。
- 小程序合法域名、HTTPS、上传版本和真机缓存是不同层次的问题，应按链路逐层验证。

## 当前成果

- 微信登录和按用户绑定网关。
- 多种 Zigbee 设备自动同步和卡片化展示。
- 插座控制、计量读取和定时任务。
- 环境数据与历史趋势。
- 彩灯亮度、颜色与色温控制。
- 小智 AI 查询与控制设备。
- 固定 HTTPS 域名、公开仓库模板、安全边界和自动测试。

## 局限与后续工作

- JSON 文件适合原型，设备和用户增加后应迁移数据库。
- 需要补充长期运行监控、日志轮转、备份恢复和进程自启动。
- 应建立设备能力模型，减少按 UIID 写条件分支。
- 历史数据可引入时序数据库并提供能耗分析和异常检测。
- 公开部署需要更严格的速率限制、Token 轮换、审计日志和隐私说明。
